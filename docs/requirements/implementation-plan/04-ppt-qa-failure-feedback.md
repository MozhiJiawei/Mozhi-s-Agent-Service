# Iteration 4: PPT QA and Failure Feedback

## Goal

Require generated PPT candidates to pass QA before they can become final
deliverables, and make failures understandable from the GitHub Issue.

## External Acceptance Feature

The Issue shows either a QA pass summary or a clear `qa_failed` status with
failure reason and retry guidance.

## Scope

- Run the required PPT QA flow after candidate generation completes.
- Collect QA results into a human-readable `qa-summary.md`.
- Update the Issue with a QA milestone comment.
- Mark the task `qa_failed` when QA rejects the candidate output.
- Mark the task `failed` when a non-QA worker or infrastructure error prevents
  completion.
- Include failure stage, concise error summary, retryability, and suggested next
  action in failure comments.
- Prevent QA-failed outputs from being treated as final archived deliverables.

## Out of Scope

- Designing new PPT quality standards beyond the existing AgentWorkspace QA
  capability.
- Archiving successful outputs.
- Publishing final download links.
- Automatic retry orchestration.

## Key Decisions

- QA is a hard gate before archive.
- `qa_failed` is distinct from `failed` because it means generation completed but
  output quality was rejected.
- QA-failed candidate files may remain in runtime scratch space for diagnosis,
  but they are not final artifacts in `briefings/`.
- Issue comments should be readable by a requester who does not have access to
  local worker logs.

## Implementation Notes

- `qa-summary.md` should describe what was checked, the result, and any notable
  limitations.
- QA pass comments should be concise and point to the summary that will later be
  archived.
- QA failure comments should avoid dumping raw stack traces unless the raw detail
  is necessary for action.
- The task state should preserve enough QA metadata for the archive step to copy
  the final summary when QA passes.
- Worker failures before QA should be marked `failed`, not `qa_failed`.

## E2E Acceptance Test

### Preconditions

- Iterations 1, 2, and 3 are complete.
- A candidate PPT can be generated by the worker.
- The AgentWorkspace QA flow is available to the worker.

### Steps

1. Submit a normal briefing request expected to generate a valid deck.
2. Open the returned Issue URL.
3. Wait for the worker to finish generation and run QA.
4. Confirm the Issue receives a QA pass summary.
5. Submit a second request with deliberately unsuitable or incomplete source
   material that should not produce a QA-approved deck.
6. Open the second Issue URL.
7. Confirm the Issue reaches `qa_failed`.
8. Confirm the failure comment includes failure stage, reason, retryability, and
   suggested next action.
9. Inspect `briefings/` and confirm the QA-failed task has no final archive
   directory.

### Expected Result

- Valid generated output proceeds past QA.
- Invalid or unacceptable output is blocked by QA.
- The requester can understand the failure from the Issue alone.
- QA-failed output is not archived as a final deliverable.

## Risks & Diagnostics

- **QA tool missing:** mark the task `failed` because QA could not run.
- **QA result ambiguous:** fail closed and require human review rather than
  archiving a questionable deck.
- **QA summary too technical:** include an operator-facing reason and a
  requester-facing summary.
- **Candidate output deleted too early:** preserve runtime paths until archive or
  failure diagnosis is complete.
- **False QA failure:** include enough retry guidance to allow a corrected source
  request or manual rerun.

## Done Criteria

- QA runs for generated PPT candidates.
- Passing QA produces a human-readable summary.
- QA failures update the Issue as `qa_failed`.
- QA-failed jobs do not create final archive directories.
